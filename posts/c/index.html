第六部分 Spring的Web MVC框架

第22章 迈向Spring MVC的旅程

Java Web的开发历程

22.1 Servlet独行天下的时代

编写响应体，输出html结构体。可以使用JavaAPI。

通过Servlet获得参数。

22.2 繁盛一时的JSP时代

不再打印html，而把需要处理的逻辑使用jsp标签，内部使用Java代码。

Servlet只需要处理数据，把数据存储到bean，然后jsp获取。并且控制jsp页面的跳转重定向。想要访问jsp必须通过Servlet匹配请求然后跳转到jsp，Servlet处理数据封装成bean传递给jsp，jsp接收数据进行页面展示。

这个时候JSPModel1。JSP+JavaBean。直接请求JSP，没有Servlet控制转发。

jsp编译之后依旧是Servlet。

22.3 Servlet与JSP的联盟

传统的MVC，web MVC。

使用多个Servlet，一个Servlet对应一个web请求。

使用单一Servlet作为集中控制器，需要自己根据URL分析处理流程的走向。

22.4 数英雄人物，还看今朝

请求驱动的web框架，基于Servlet的请求响应。SpringMVC。

事件驱动的web框架，基于视图中的某些组件。点击事件

第23章 Spring MVC初体验

23.1 鸟瞰Spring MVC

Front Controller（接收处理所有web请求） + Page Controller（具体请求的具体处理逻辑）。

Servlet主要获取请求信息、调用业务对象进行处理、将模型数据传递给视图并返回视图。

  1. HandllerMapping，（根据web请求找到对应的page controller）

    不同的HandllerMapping实例可以有不同的匹配规则，可以扩展。而匹配规则在DispatcherServlet中写死则不能扩展，所以创造了HandllerMapping。三种匹配方式如下：

    1. 掐头去尾（去除上下文路径context-path和.html等后缀），然后匹配得到controller
    2. 以请求参数的值，如：url?controller=HelloController
    3. 以cookie或session的某些信息进行匹配

  2. Controller（web请求的具体处理者）

    1. HandllerMapping返回的结果是Controller，
    2. Controller处理完返回一个ModelAndView实例（包含模型数据和视图的逻辑名称）

  3. ViewResolver和View（ViewResolver根据视图名找到视图文件）

    ViewResolver类似于HandllerMapping处理映射（逻辑视图名与View实例），View类似于Controller。

    1. 视图技术不止有jsp，使用ViewResolver和View可以屏蔽差异性（经相同的模型数据纳入不同的视图中）。
    2. 向客户端输出视图结果（jsp等）有两种方式，文本、二进制
      1. httpServletResponse.getWriter
      2. httpServletResponse.getOutputStream
    3. View的实现类负责渲染视图，将模型数据合并到视图模板中，并把视图结果返回输出到客户端。

23.2 实践出真知

23.2.1 Spring MVC应用的物理结构

目录结构和部署描述符文件（用于描述项目部署的元信息的文件），

  1. ContextLoaderListener与/WEB-INF/applicationContext.xml（最顶级的Spring配置文件）
    1. web.xml，<listener>标签，ServletContextListener（子类有一个ContextLoaderListener），可以监听容器初始化和销毁
    2. ContextLoaderListener职责是为整个Web应用加载顶层的WebApplicationContext（一种web专用的应用容器，类似于ClassPathXmlApplicationContext和FileSystemXmlApplicationContext）
    3. ContextLoaderListener 加载 WebApplicationContext的默认配置文件的位置就是/WEB-INF/applicationContext.yml。
    4. 若加载多个配置文件，可以在web.xml中更改默认位置，配置context-参数contextConfigLocation。逗号和空格可以分割多个文件路径。
    5. ContextLoaderServlet 和 ContextLoaderListener 相同的工作，不过要更改load-on-startup的值，让它在其他的Servlet之前启动。
    6. WebApplicationContextUtils工具类可以获取绑定到ServletContext的 WebApplicationContext
  2. DispatcherServlet与XXX-Servlet.xml（SpringMVC配置文件，父容器是applicationContext.xml）
    1. DispatcherServlet有一个外部配置文件（SpringMVC配置文件），配置HandlerMapping、Controller、ViewResolver等，默认路径也是/WEB-INF/，文件名是XXX-Servlet.xml，xxx是webxml中DispatcherServlet定义的servlet-name。
    2. DispatcherServlet加载配置文件之后，会构建一个wac（WebApplicationContext），该wac会把ContextLoaderListener构建的wac（ROOT DispatcherServlet）作为父容器。
    3. webxml，DispatcherServlet 的 contextConfigLocation初始化参数可以指定多个配置文件。（第1条的第4点）。
23.2.2 按部就班地开始工作
  1. 建立工程
    1. 配置web.xml。
      • DispatcherServlet，<servlet>。ContextLoaderListener，<listener>。
      • Filter，ServletContextListener等，处理字符编码和 log4j 初始化
    2. WebApplicationContext文件的添加
      • /WEB-INF/applicationContext.xml（最顶级的Spring配置文件），注册中间层服务。业务Service的Bean也在这里注册。
      • [servler-name]-Servlet.xml（SpringMVC配置文件）
  2. 开发独立的业务逻辑。
  3. 添加web请求入口。a标签跳转页面。
  4. 添加HandlerMapping。请求url 映射到 controller。默认是 BeanNameUrlHandlerMapping，自定义注册是在 [servler-name]-Servlet.xml 文件中。
  5. 实现对应的Controller并添加到配置文件。在 [servler-name]-Servlet.xml 文件中声明 controller 的bean。BeanNameUrlHandlerMapping要求controller的bean名以/开头
  6. 添加ViewResolver。根据视图名找到对应的视图文件，InternalResourceViewResolver处理JSTL。
  7. 实现相应视图。创建视图文件如jsp。

第24章 近距离接触Spring MVC主要角色

24.1 忙碌的协调人HandlerMapping

HandlerMapping 映射URL到具体的处理类的匹配

次级控制器不止Controller，也可以是其他类型。

次级控制器在SpringMVC中成为Handler。

HandlerMapping 是一个接口，返回 HandlerExecutionChain 类型（包含了具体的Handler）。

24.1.1 可用的HandlerMapping

前面讲过 BeanNameUrlHandlerMapping。直接配置路径，强制了bean的定义名称匹配连接路径。

还有SimpleUrlHandlerMapping、controllerClassNameHandlerMapping 、DefaultAnnotationHandlerMapping。

  1. SimpleUrlHandlerMapping，可以使用类似ant匹配。
24.1.2 HandlerMapping执行序列(Chain Of HandlerMapping)

HandlerMapping有多个，可以设置优先级（Ordered接口），值小优先级大，默认最大值。

24.2 我们的亲密伙伴Controller

Controller接口，实现一个Controller可以直接实现接口。

Controller 分为两派：

  1. 自由挥洒派（AbstractController 、MultiActionController ）。比较自由，近似于Servlet，可以自己通过HttpServletRequest获取参数、HttpServletResponse输出视图。
  2. 规范操作派（SimpleFormController 、AbstractWizard-FormController 等）。进行封装，统一抽取参数并封装到对象。统一的数据验证方式。
24.2.1 AbstractController

Controller接口的直接抽象子类，使用了模板方法模式，有如下关注点

  1. 管理当前Controller支持的请求类型（get、post等）
  2. 管理页面的缓存设置（是否允许浏览器缓存当前页面）
  3. 管理执行流程在会话上的同步

而我们需要做的，继承，然后重写 handleRequestInternal(HttpServletRequest var1, HttpServletResponse var2) 方法，返回ModelAndView类型。

24.2.2 MultiActionController

继承了AbstractController ，

  1. 请求参数到Command对象的绑定
  2. 通过Validator的数据验证
  3. 细化的异常处理方法

MethodNameResolver，策略接口，主要用来帮助MultiActionController 决定当前web请求交给哪个方法解决，会返回一个方法名。

可用的MethodNameResolver 实现，

  1. InternalPathMethodNameResolver。（默认）

    1. 路径的最后一段，去掉后缀去掉前面的/，就是方法名。
    2. 比如：/test/macontroller/creat.do，对应方法creat。
    3. 也可以指定前缀后缀，若上面例子指定前缀rate_，对应方法 rate_create

  2. PropertiesMethodNameResolver。

    1. 与 InternalPathMethodNameResolver类似，只不过可以使用Ant表达式匹配多个。

  3. ParameterMethodNameResolver。

    1. 根据请求中的某个参数作为方法名，默认参数名是action，也可以通过parameterName配置。?action=methodName&其他参数。

    2. 根据请求中的一组参数作为方法名，其中的任意一个都可以，但是若同时存在两个则不行。

      defaultMethodName参数可以设置一个默认处理方法。

      若同时存在methodParameterNames 和 parameterName，则methodParameterNames 优先级高。
