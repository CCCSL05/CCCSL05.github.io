第五部分 事务管理

第17章 有关事务的楔子

17.1 认识事务本身

并发访问数据的时候，需要事务来保证系统前后的逻辑正确性。

事务有ACID4个特征：

  1. 原子性【事务本身】

    原子操作，执行语句要么全部完成，要么全部回退回去。

  2. 一致性【系统本身】

    银行AB转钱。

  3. 隔离性【事务之间】

    4种隔离级别：

    1. Read Uncommitted，可以读到未提交的数据
    2. Read Committed【默认】，读到是已经提交过的数据
    3. Repeatable Read，读到的数据不允许其他食物更改。
    4. Serializable，事务串行化，一个执行完，另一个才能执行
    • 脏读【更改后被读取到了】，读到另一个未提交的事务执行期间更新的数据。
    • 不可重复读【读取后被更改了，再读取】，读到另一个未提交的事务执行期间多次更新的数据。
    • 幻读【数据增多减少】，读到另一个未提交的事务执行期间新增或删除的数据，比如读整体数量的时候。

    默认隔离级别大多是Read Committed，Hsqldb是Read Uncommitted。

    并非所有数据库都支持这四种隔离级别，如Oracle支持2和4。

  4. 持久性【系统本身】

    不会被回退。修改不会丢失。

17.2 初识事务家族成员

事务的几个参与者：

  1. Resource Manager，管理数据资源，如数据库服务器
  2. Transaction Processing Mointor，事务过程监视器。协调多个RM的处理
  3. Transaction Manager，事务管理器。
  4. Application

全局事务，多个RM。

局部事务，一个RM。

第19章 Spring事务王国的架构

使用统一的编程模型来进行事务编程。

设计理念：让事务管理的关注点与数据访问的关注点相分离

19.1 统一中原的过程

  • 事务抽象的顶层接口PlatformTransactionManager，为应用程序提供统一的事务界定。

    三个方法，getTransaction，commit，rollback。

  • 要保证两个DAO的数据访问方法处于同一事务，就得保证它们是同一个Connection。

  • 把Connection绑定到当前线程上，从线程上获取，提交或回滚之后解除绑定。

  • 实现PlatformTransactionManager接口，重写方法，获取连接时绑定连接TransactionResourceManager中，提交或回滚时再解绑。

  • 实际上，DataSourceUtils主要的工作是对Connection的管理，获取con时如果没有会从DataSource那里引用新的con，如果有就返回。

19.2 和平年代

PlatformTransactionManager接口界定事务边界，根据事务状态开启事务。有两个子接口：

  1. TransactionDifinition，事务相关属性定义，包括隔离级别、传播行为等。
  2. TransactionStatus，事务状态。
19.2.1 TransactionDefinition

1 TransactionDefinition 简介

定义了事务的相关属性，隔离级别、传播行为、超时时间、是否为只读事务。

隔离级别有5个常量：

  1. ISOLATION_DEAULT，默认是read commited。
  2. ISOLATION_READ_UNCOMMITED
  3. ISOLATION_READ_COMMITED
  4. ISOLATION_REPEABLE_READ
  5. ISOLATION_SERIALIZABLE

事务的传播行为：

运用场景//todo

  1. PROPAGATION_REQUIRED【默认】，需要一个事务保持运行。有的话就加进去，没有的话就创建一个。
  2. PROPAGATION_SUPPORTS，支持事务中运行。如果有就加入，没有就直接运行。常用于查询方法。
  3. PROPAGATION_MANDATORY，mandatory强制的，当前必须有一个事务，否则会抛异常。该方法必须在事务内运行。
  4. PROPAGATION_REQUIRES_NEW，创建一个新的事务，如果已经有事务则把旧事务挂起。常用于当前事务成功与否不影响到整体。
  5. PROPAGATION_NOT_SUPPORT，不支持事务，当前没有事务的时候才会运行，有事务则会被挂起。
  6. PROPAGATION_NEVER，不需要事务，当前存在事务则会抛异常。
  7. PROPAGATION_NESTED，nested嵌套，当前有事务旧嵌套进去，没有就创建一个新的事务。并非都支持这个行为。

TIMEOUT_DEFAULT常量，指定事务的超时时间。默认-1。

是否只读，只读的话，通过TransactionDefinition 的实现类设置。默认false。

2 TransactionDefinition 相关实现

按编程式和声明式场景分为两个分支，DefaultTransactionDefinition 、TransactionAttribute。

  1. DefaultTransactionDefinition 是TransactionDefinition 的默认实现类，定义了一些属性的默认值，并且定义了相关setter。

    TransactionTemplate继承自DefaultTransactionDefinition，编程式事务，

  2. TransactionAttribute，继承自TransactionDefinition（额外添加了rollbackOn回滚方法），主要面向使用AOP进行声明式事务管理，

    DefaultTransactionAttribute，继承自DefaultTransactionDefinition，实现了TransactionAttribute。有两个实现类：

    1. RuleBasedTransactionAttribute，可以指定多个回滚规则，用于rollbackOn方法。
    2. DelegatingTransactionAttribute，抽象类，被子类化。调用委托给另一个TransactionAttribute实现类。

事务处理过程中的状态。

可以查询事务状态，setRollbackOnly，保存点回滚。

第20章 使用Spring进行事务管理

20.1 编程式事务管理

有两种方式：

  1. PlatformTransactionManager（这个是接口）
  2. 更方便的TransactionTemplate（模板类，推荐）
20.1.1 直接使用PlatformTransactionManager进行编程式事务管理
  1. 提供PlatformTransactionManager的实现类
  2. 结合 TransactionDefinition 开启事务
  3. 并结合 TransactionStatus 回滚或提交事务，

完全控制整个事务的处理过程，过于底层，流程固定。可以使用模板方法模式和Callback回调接口封装成TransactionTemplate。

20.1.2 使用TransactionTemplate进行编程式事务管理

类似于JDBCTemplate，

  • 有两个Callback接口，TransactionCallback（有处理结果），TransactionCallbackWithoutResult（没有返回处理结果）

  • 回滚的两种方式：1抛出unchecked异常，2TransactionStatus（Callback方法参数）标记事务状态为rollBackOnly（提交前会检查状态）。

  • TransactionTemplate无法处理向上抛出checked exception的这种情况

20.1.3 编程创建基于Savepoint的嵌套事务

使用 TransactionStatus 创建基于Savepoint的嵌套事务。

使用 TransactionStatus 可以创建savepoint，然后调用相关方法可以回滚到这个保存点。最后释放保存点。

20.2 声明式事务管理

20.2.1 引子

  1. 事务管理的代码从业务对象种剥离出来，动态代理（可以获得原对象的业务方法，然后自身处理事务）。

  2. 使用SpringAOP，提供一个拦截器，在业务方法之前开启事务，方法完成后提交事务，方法异常回滚事务。

  3. 声明式事务管理需要记录驱动事务的元数据（比如什么异常回滚，事务级别，传播行为）

    • 两种方式：使用XML记录，或者使用注解记录。

    • TransactionAttribute 是 TransactionDefinition 实现类，多一个什么异常才会回滚的信息，所以使用TransactionAttribute代替默认的 TransactionDefinition 实现类。

20.2.2 XML元数据驱动的声明式事务

从spring 1.x到2.x，有4种配置方式在ico的配置文件中指定元数据

  1. 使用 ProxyFactory（ProxyFactoryBean）+TransactionInterceptor

    1. 本质上，是为TransactionInterceptor提供需要的元数据信息（TransactionDefinition，具体应该是TransactionAttribute）

      TransactionInterceptor 实现 MethodInterceptor接口，实现invoke方法。动态代理拦截所有方法（可通过方法名判断过滤）

    2. 需要，datasource，jdbcTemplate，transactionManager，拦截器，ProxyFactoryBean，业务对象

    3. 拦截器拦截匹配的业务方法进行aop操作，拦截器相当于Aspect

    4. 元数据TransactionAttribute可以通过TransactionAttributeSource接口的方法获得，所有有一个TransactionAttributeSource类型的成员，不同的实现类有不同的方式存储信息。

    5. xml里，TransactionInterceptor的元数据可通过TransactionAttribute属性，也可以通过TransactionAttributeSource属性（全限定方法名）。

  2. 使用一站式的TransactionProxyFactoryBean（上一种的简化方式）

    1. 把TransactionInterceptor的属性，放在了TransactionProxyFactoryBean里，可以直接设置
    2. 简化：不设置target属性（抽象bean）。通过parent继承，然后两个bean设置不同的target值

  3. 使用 BeanNameAutoProxyCreator（记录业务对象的beanName）

    1. BeanNameAutoProxyCreator做了ProxyFactoryBean的工作，有一个TransactionInterceptor属性，和一个beanNames属性记录业务对象的beanName列表（target列表）。
    2. 本质上是为beanNames列表的业务对象 动态代理生成代理对象

  4. 使用spring2.x声明配置方式

    1. 引入tx命名空间，<tx:advice>底层是TransactionInterceptor
    2. <tx:attribute>记录元数据，子标签是<tx:method >
    3. <aop:config>配置底层是自动代理
20.2.3 注解元数据驱动的声明式事务

直接将注解@Transactional标注到业务对象的方法上或业务对象上。通过反射获取元数据。

在xml里声明 <tx:annotation-driven >就可以自动处理注解。

推荐将注解标注到具体的业务实现类/类方法 上，因为AOP有两种方式生成代理对象，而CGLIB无法读取接口上的注解信息（字节码层面）。通过接口的动态代理可以（编译前，代码层面，可以反射）。

第21章 Spring事务管理之扩展篇

21.1 理解并活用ThreadLocal

21.1.1 理解ThreadLocal的存在背景

一个对象被3个线程同时访问，要么线性处理（1对3），或者变成3个对象（3对3），这就是ThreadLocal存在的意义，每一个线程都有自己专属的对象，不会影响到其他线程的对象状态。

多线程对共享对象的影响。

21.1.2 理解ThreadLocal的实现
  • Thread ThreadLocal和ThreadLocalMap，用法+原理，我懵圈了？ - 知乎 (zhihu.com)

  • 每个Thread线程内部都有一个ThreadLocalMap。
  • Map里面存储线程本地对象ThreadLocal（key）和线程的变量副本（value）。
  • Thread内部的Map是由ThreadLocal维护，ThreadLocal负责向map获取和设置线程的变量值。
  • 一个Thread可以有多个ThreadLocal。

aThreadLocal在3个线程中分别对应abc

bThreadLocal在3个线程中分别对应123

21.1.3 ThreadLocal的应用场景

为多线程分配各自的数据资源，不共享。

  1. 有状态而非线程安全的对象，使用ThreadLocal。有状态意味着不能被共享改变。比如connection连接（事务）。
  2. ThreadLocal在单一线程内传递数据（时间前后），单一线程的全局存储。
  3. 本来就是多线程共享的对象设置ThreadLocal提高效率，以空间换时间
  4. 每个线程单例。某些资源初始化代价大，可以初始化之后放到ThreadLocal。
21.1.4 使用ThreadLocal管理多数据源切换的条件

通过ThreadLocal保存数据源对应的标志。AbstractRoutingDataSource，determineCurrentLookupKey。

可以使用AOP切换数据源。
