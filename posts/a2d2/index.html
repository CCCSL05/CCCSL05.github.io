七大原则

1、开闭原则

‘开’是可以增加新的代码，‘闭’是不去修改已有的代码，从而实现需求和功能。

例如：抽象类新增一个新的实现类。抽象主题Theme接口 可以新增一个夜间主题的实现类，而不去修改日间主题的代码。

2、里氏替换原则

里氏代换原则中说，任何基类可以出现的地方，子类一定可以出现。

里氏替换原则通俗来讲就是：子类可以扩展父类的功能，但不能改变父类原有的功能。

3、依赖倒置原则

高层模块不应该依赖低层模块，两者都应该依赖其抽象；抽象不应该依赖细节，细节应该依赖抽象。

其核心思想是：要面向接口编程，不要面向实现编程。面向接口写实现，而不是面向实现写接口。

4、单一职责原则

一个类负责一项职责（功能）。

这里的职责是指类变化的原因，单一职责原则规定一个类应该有且仅有一个引起它变化的原因，否则类应该被拆分。

5、接口隔离原则

建立单一接口，不要建立庞大臃肿的接口，尽量细化接口，接口中的方法尽量少。

将广义上的接口 具体成狭义上的接口。

例如，小说接口 可具体成 言情小说、玄幻小说、仙侠小说…读者需要哪一种可自行实现对应的更为具体接口

6、迪米特法则

又称最少知道原则，一个实体应当尽量少地与其他实体之间发生相互作用，使得系统功能模块相对独立。

如果两个软件实体无须直接通信，那么就不应当发生直接的相互调用，可以通过第三方转发该调用。其目的是降低类之间的耦合度，提高模块的相对独立性。

但是，过度使用迪米特法则会使系统产生大量的中介类，从而增加系统的复杂性，使模块之间的通信效率降低。所以，在釆用迪米特法则时需要反复权衡，确保高内聚和低耦合的同时，保证系统的结构清晰。

7、合成复用原则

又称组合/聚合复用原则。

尽量使用组合和聚合少使用继承的关系来达到复用的原则。

8、一句话总结（表格）

设计原则 一句话归纳 目的
开闭原则 对扩展开放，对修改关闭 降低维护带来的新风险
里氏替换原则 不要破坏继承体系，子类重写方法功能发生改变，不应该影响父类方法的含义 防止继承泛滥
依赖倒置原则 高层不应该依赖低层，要面向接口编程 更利于代码结构的升级扩展
单一职责原则 一个类只干一件事，实现类要单一 便于理解，提高代码的可读性
接口隔离原则 一个接口只干一件事，接口要精简单一 功能解耦，高聚合、低耦合
迪米特法则 不该知道的不要知道，一个类应该保持对其它对象最少的了解，降低耦合度 只和朋友交流，不和陌生人说话，减少代码臃肿
合成复用原则 尽量使用组合或者聚合关系实现代码复用，少使用继承 降低代码耦合

创建型

1、单例模式（6种实现）

1、单例类只能有一个实例。

2、单例类必须自己创建自己的唯一实例。（构造函数私有）

3、单例类必须给所有其他对象提供这一实例。（通过提供public方法返回实例对象）

（1）懒汉式-线程不安全

public class Singleton {  
    private static Singleton instance;  
    private Singleton (){}  
  
    public static Singleton getInstance() {  
    if (instance == null) {  
        // 多线程可能同时判断为null，都进来创建实例
        instance = new Singleton();  
    }  
    return instance;  
    }  
}

（2）懒汉式-线程安全

public class Singleton {  
    private static Singleton instance;  
    private Singleton (){}  
    // 方法加锁保持同步，但加锁会影响效率
    public static synchronized Singleton getInstance() {  
    if (instance == null) {  
        instance = new Singleton();  
    }  
    return instance;  
    }  
}

（3）饿汉式-线程安全

public class Singleton {  
    // 速度快。类加载时就初始化，浪费内存。
    private static Singleton instance = new Singleton();  
    private Singleton (){}  
    public static Singleton getInstance() {  
    	return instance;  
    }  
}

（4）双重校验锁-线程安全

public class Singleton {  
    private volatile static Singleton singleton;  
    private Singleton (){}  
    public static Singleton getSingleton() {  
        if (singleton == null) {  
            // （1）懒汉式-线程不安全 升级版。
            synchronized (Singleton.class) {  
                if (singleton == null) {  
                    singleton = new Singleton();  
                }  
            }  
        }  
    	return singleton;  
    }  
}

（5）静态内部类实现

public class Singleton {  
    // 静态内部类
    private static class SingletonHolder {  
        // 类变量 隶属类，是单例
    	private static final Singleton INSTANCE = new Singleton();  
    }  
    private Singleton (){}  
    public static final Singleton getInstance() {  
        // 此时调用内部类，才会加载内部类，才会初始化 INSTANCE 实例
    	return SingletonHolder.INSTANCE;  
    }  
}

（6）枚举实现

public enum Singleton {
    uniqueInstance;
}

2、简单工厂（一个大工厂）

具体工厂-具体产品：一对多

简单工厂，根据传递过来的参数值创建对应的产品，所有产品都由简单工厂创造，创造饮料、服装等，所以 添加产品的话要修改代码(违背开闭原则)。

3、工厂方法（分类子工厂）

具体工厂-具体产品：一对一

把工厂抽象化，创造产品交给具体的子工厂（相当于简单工厂），添加产品的话只需要添加其对应的子工厂就行啦（这里有多个简单工厂）。小说抽象工厂，玄幻小说具体工厂只生产玄幻小说，言情小说具体工厂只生产言情小说。

从抽象工厂获取具体工厂。再从具体工厂获取具体产品

4、抽象工厂（家族工厂）

具体工厂-具体产品：多对多

具体工厂2个：小米工厂，苹果工厂。

抽象产品2种：手机（具体分为小米手机和苹果手机），自行车（小米自行车和苹果自行车）

5、生成器（建造者模式）

一个整体（对象）是由各个子部件（属性）构成的，我们可以指定子部件分别是什么，从而得到我们想要的整体对象。

一个页面分为上中下三个部分，我们可以分别为三个部分指定颜色，从而得到我们想要的整体效果。构造器（建造者）builder.setShang(“颜色”).setZhong(“颜色”).setXia(“颜色”)

6、原型模式（clone）

原型模式是根据原型再克隆出一个类似或相同的对象，节省资源、成本。只要实现了Clone able接口就是原型模式

根据原型实例，通过克隆(实现Cloneable接口)创建相同或相似的对象

  • 克隆分为深克隆(会递归调用子对象的clone方法) 和 浅克隆
  • 为了实现深克隆，每一层对象对应的类都必须支持深克隆

结构型

结构型模式，分为类结构型模式和对象结构型模式，前者采用继承机制来组织接口和类，后者釆用组合或聚合来组合对象。

1、外观（一键挂机）

在现实生活中，常常存在办事较复杂的例子，如办房产证或注册一家公司，有时要同多个部门联系，这时要是有一个综合部门能解决一切手续问题就好了。

软件设计也是这样，当一个系统的功能越来越强，子系统会越来越多，客户对系统的访问也变得越来越复杂。这时如果系统内部发生改变，客户端也要跟着改变，这违背了“开闭原则”，也违背了“迪米特法则”，所以有必要为多个子系统提供一个统一的接口，从而降低系统的耦合度，这就是外观模式的目标。

一键刷怪。

刷怪要去访问很多子接口，流程繁琐。如果有一个挂，填好相关数据，点击一下就可以自动刷怪。那么这个挂我们一般称之为“外观模式”。当子接口改变了，我们不用担心，由开发外挂的人维护就行了。我们还是只需要打开外挂填好数据就行了。

2、适配器（读卡器）

讲中文的人同讲英文的人对话时需要一个翻译，用直流电的笔记本电脑接交流电源时需要一个电源适配器，用计算机访问照相机的 SD 内存卡时需要一个读卡器（适配器），为了使目标能够使用适配者（为了使计算机能够使用内存卡）。

适配器就是转换器。

类的适配器，用一个子类实现目标接口，创建目标对象直接调用方法。

类的适配器模式是用多重继承（Java不支持），而对象的适配器模式是用组合。

// 将一个A接口转换为B接口，使得新的对象符合B接口规范。
public BAdapter implements B {
    private A a; // 对象的适配器模式，使用组合
    public BAdapter(A a) {
        this.a = a;
    }
    public void b() {
        a.a();
    }
}
B b = new BAdapter(a); // 给适配器一个A对象，返回一个B对象
b.b(); // B对象调用A的方法

3、桥接（n维向量）

桥接模式通过分离一个抽象接口和它的实现部分，使得设计可以按两个维度独立扩展。

颜色，形状，两个接口。实现不同颜色的不同形状。

创建不同颜色，再创建不同形状，组合颜色对象，把颜色对象拿过来（套进来，就像套娃一样）

颜色是一维，形状包含颜色是二维，大小包含形状是三维，可以无限套娃…

把你的色我的形，串一串，串一个新对象，串一个特征向量。

// 不同颜色、大小的形状，其中颜色大小都是修饰（形容，描述）Shape实体对象的

// 抽象类，实现类可以有圆，方形，三角形
public abstract class Shape {
    Color color; // 抽象类，实现类可以有 红、黄、蓝
 
    public void setColor(Color color) {
        this.color = color;
    }
    
    //Size size; // 抽象类，实现类可以有 大中小
    //public void setSize(Size size) { // 桥接另外的接口（修饰作用的），Shape是实体
    //    this.szie = size;
    //}
    
    public abstract void draw();
}

4、组合（树形结构）

将对象组合成树形结构来表示“整体/部分”层次关系，允许用户以相同的方式处理单独对象和组合对象。

组合模式（Composite Pattern），又叫部分整体模式，是用于把一组相似的对象当作一个单一的对象。组合模式依据树形结构来组合对象，用来表示部分以及整体层次。

树形结构。员工（包含下级直属员工）。N维数组。N维层次

5、装饰（机甲）

装饰器（Decorator）模式的定义：指在不改变现有对象结构的情况下，动态地给该对象增加一些职责（即增加其额外功能）的模式，它属于对象结构型模式。增加了额外功能，但没有使用继承。

人，机甲。把人对象给机甲对象，从而扩展了人对象的功能。

人不一定需要机甲，而机甲一定需要人作为载体。所以机甲是装饰品，可要可不要。

6、享元

利用共享的方式来支持大量细粒度的对象，这些对象一部分内部状态是相同的。 它让某个类的一个实例能用来提供许多”虚拟实例”

享元要求实例不能变化但可以创建多个对象。而单例要求实例只创建一个但可以变化。

享元（Flyweight）的核心思想很简单：如果一个对象实例一经创建就不可变，那么反复创建相同的实例就没有必要，直接向调用方返回一个共享的实例就行，这样即节省内存，又可以减少创建对象的过程，提高运行速度。

在实际应用中，享元模式主要应用于缓存，即客户端如果重复请求某些对象，不必每次查询数据库或者读取文件，而是直接返回内存中缓存的数据。

7、代理

由于某些原因需要给某对象提供一个代理以控制对该对象的访问。

这时，访问对象不适合或者不能直接引用目标对象，代理对象作为访问对象和目标对象之间的中介。

// 与适配器模式类似，把A接口转换为A接口
public AProxy implements A {
    private A a;
    public AProxy(A a) {
        this.a = a;
    }
    public void a() {
        // 在方法执行前添加额外功能
        this.a.a();
        // 在方法执行后添加额外功能
    }
}
