二、java

2.1 Java基础知识

2.1.1 重载和重写的区别

重载： 同一个类，方法名相同，参数列表不同（参数类型，个数，顺序），返回值和访问修饰符随意。

重写： 父子类，方法签名相同，返回值比父类更详细，抛出的异常更详细，访问范围大于等于父类，这是因为当子类替代父类时，原有代码不受影响。返回值类型可以自动转换，异常可以被处理，对象可以访问。

Definition: Two of the components of a method declaration comprise the method signature—the method’s name and the parameter types.

方法声明的两个组件构成了方法签名 - 方法的名称和参数类型。

2.1.2 String StringBuffer StringBuilder

字符串底层是一个字符数组

  1. 可变性
    • String 不可变，因为底层代码是 private final char value[]，使用了 final 修饰
    • 而 StringBuffer StringBuilder 没有使用 final 修饰，故可变
  2. 线程安全性
    • String 不可变，是一个常量，每一个线程访问的都一样，故线程安全
    • StringBuffer 内部使用了 Synchronize 加了同步锁，故实现线程安全
    • 而 StringBuilder 没有加同步锁，所以线程不安全
  3. 性能
    • String 另外创建了一个新的对象，故效率最低，适合操作少量数据
    • StringBuffer StringBuilder 都适合操作大量数据。因为 StringBuffer 使用了同步锁，故效率比 StringBuilder 稍微差一点（ 10%~15%）。
    • 操作大量数据，单线程使用 StringBuilder ，多线程使用 StringBuffer

2.1.3 自动装箱与拆箱

装箱：基本类型 -> 包装类型

拆箱：包装类型 -> 基本类型

Integer x = 2;     // 装箱
int y = x;         // 拆箱

// 编译之后如下，像是语法糖，简化了开发
Integer x = Integer.valueOf(2);     // 装箱
int y = x.intValue();               // 拆箱

2.1.4 == 与 equals

  • == 是关系运算符，比较两个变量的值是否一样，或是比较两个对象的引用是否一样（存储的是对象在堆中的地址）
  • equals 是 Object 的方法，每一个类都是 Object 的子类，故每个对象都有 equals 方法。方法返回 boolean 类型，可被重写，自定义比较规则。若不重写，则比较的是两个对象的引用是否相同。

2.1.9 接口和抽象类的区别是什么

  • 抽象类，包含抽象方法（也可以有非抽象方法），不能创建抽象类对象，其他与普通的类别无二致
  • 接口类，成员变量只能是公开的类常量（public static final）在创建时必须初始化，没有构造方法，方法默认用 public abstract 修饰，没有静态方法。（另，java8接口有默认方法）
  • 一个类可以实现多个接口，但只能继承一个抽象类。一个非抽象类，实现了接口或者继承了抽象类，要实现所有的抽象方法。
  • 抽象类，像是类的模板。接口，像是一组行为规范。

2.2 Java 集合框架

2.2.1 Arraylist 与 LinkedList 异同

  1. 是否线程安全
    • 二者都可变，且没有进行同步。故线程不安全
  2. 底层数据结构
    • Arraylist 底层是 Object 数组
    • LinkedList 底层是 双向链表（首尾不相连），1.6 之前为双向循环链表
  3. 插入和删除是否受元素位置影响
    • Arraylist 是数组，插入和删除都受影响
    • LinkedList 是链表，不受影响
  4. 快速随机访问
    • Arraylist 是数组，支持快速随机访问
    • LinkedList 是链表，不支持快速随机访问
  5. 内存空间占用
    • Arraylist 是数组，尾部会有空间没有使用
    • LinkedList 是链表，每个元素要多占用空间（记录直接前驱和直接后继）
  6. 如何扩容，默认初始空间大小
    • ArrayList初始大小为10，扩容后为 1.5n+1
    • LinkedList是链表，没有初始大小，也没有扩容机制，就是通过指针连接添加一个新的节点而已

2.2.2 ArrayList 与 Vector 区别

  • Vector 类的所有方法都是同步的，保证了线程安全，但是单线程很耗时间很慢
  • ArrayList 线程不安全，但是快

2.2.3 HashMap的底层实现

  1. 1.7 链表散列（即 数组链表相结合）
  2. 1.8 链表散列（链表长度>8转红黑树）

2.2.4 HashMap 和 HashTable的区别

  1. 线程是否安全
    • HashTable 的方法使用了Synchronize同步，所以线程安全，所以平时不用它
    • HashMap 线程不安全，所以经常使用。
  2. 效率
    • HashMap 线程不安全，所以HashMap 比HashTable稍微快一点
  3. 底层数据结构
    • 都是链表散列，只不过HashMap在1.8之后可以转红黑树
  4. 对null key 和 null value 的支持
    • HashMap中可以有一个key为null，多个v为null
    • HashTable中kv都不能为null，有一个为null就会抛空指针异常
  5. 初始大小 和 每次扩容大小
    • HashMap 初始大小16，扩容变成原来的 2^n
    • HashTable 初始大小11，扩容变成原来的 2n+1

2.2.5 HashMap 的长度为什么是2的幂次方

  • 为了hash运算更快
  • 因为 正常取模%运算非常慢，效率不如位运算，又因为有这么一个公式在除数(即length)是2的幂次方时成立 hash%length==hash&(length-1)，所以当length是2的幂次方时，取模运算可以变成&与运算，从而更快得到散列值。

3.5 TCP 三次握手和四次挥手

  1. 三次握手
    • 双方确认自己和对方的发送和接受是正常的（双方都可以正常发送接收）。
  2. 四次挥手
    • 第一次，A：我没啥好说的了，我想要要关闭连接了
    • 第二次，B：我知道你的想法了，但是我还有些东西没说完，你再等一会。
    • 第三次，B：我的话说完了，你可以关闭了。（等待两个MSL，若没有收到回信，则超时重发。收到了就关闭）
    • 第四次，A：好，我准备关了，你收到消息也关闭吧。（此时A并没有立即关闭，要等两个MSL，万一B没收到，B会再次发送第三次的那个请求）

五、MySQL

5.1 存储引擎

存储引擎：

MySQL 有多种存储引擎，每种引擎下的表结构、事务操作、存储机制…有所不同，适用于不同的读写场景。

InnoDB类型的表，MyISAM类型的表。

  1. InnoDB
    • MySQL 5.5 改为默认的事务型存储引擎
    • 支持事务，而 MyISAM 不支持
    • 支持外键，而 MyISAM 不支持
    • 支持行级锁，而 MyISAM 是表级锁
    • 支持在线热备份
    • 崩溃后的安全恢复好
  2. MyISAM
    • MySQL 5.1 及之前的默认引擎，5.1 下一个就是 5.5
    • 占用空间小，访问速度快，更适合读，而InnoDB适合写

5.2 索引

5.2.1 为什么索引能提高查询速度?

  • 底层数据结构是 B+ 树，提高了查询速度（不用一个一个遍历查询）。
  • 索引将无序的数据变得相对有序。（就像目录，从全表扫描变成查目录）

5.2.2 聚簇索引 和 非聚簇索引

  1. 聚簇索引（主索引）
    • 该索引下的列值的逻辑顺序（1 2 3、A B C）与物理存储顺序相同。（新华字典的侧面黑方块是A-Z的顺序，内容也是A-Z的顺序排列的，直接按黑方块查找，找到了索引也就找到了数据）
    • 也叫 主键索引，一个表只能有一个。
  2. 非聚簇索引（辅助索引）
    • 列值的逻辑顺序和该记录的物理存储顺序 不相同的都叫 非聚簇索引
    • 也叫 二级索引。查完目录，再查内容（新华字典的目录是A-Z顺序排列的，内容也是A-Z的顺序排列的）

5.2.3 最左匹配原则

联合索引（a,b），查询条件有ab或a时都可以触发索引，但是只有一个b却不能使用索引加速。

也就是说只有b左边的索引全部有值，b才能使用索引加速。

5.4 事务

一组sql操作集合，要么都执行要么都不执行。

5.4.1 事务的特性（ACID）

  1. Atomicity（原子性）
    • 事务是一个整体，事务里的所有操作要么全完成，要么全不完成。
  2. Consistency（一致性）
    • 事务前后，数据库的完整性没有被破坏，是逻辑正确的，即对的上账，
  3. Isolation（隔离性）
    • 事务操作时会对数据进行读写，并发事务存在多个事务同时操作数据，可能导致数据不一致。因此需要事务隔离，以便正常的完成业务。
    • 事务有四个隔离级别，防止事务之间的干扰
  4. Durablity（持久性）
    • 事务提交后，数据的修改是永久的，即使系统故障也不会丢失。

5.4.2 并发事务一致性问题

如果没有事务隔离，会出现以下问题

  1. 丢失修改
    • 修改覆盖，T1刚改完，T2就又改了
  2. 脏读
    • T1修改了数据，但没有提交。T2读取了这个数据并使用了，最后提交了。
    • T2成功的事务使用了失败的事务修改的数据
    • 读到的数据是一个假的数据
  3. 不可重复读
    • T1修改数据，T2在修改前后读取使用
    • T2要读取使用的数据被T1修改
    • 前后读到的数据不一样
  4. 幻读
    • T1新增数据，T2在新增前后读取使用
    • T2要读取使用的数据整体发生变化
    • 前后读到的数据整体不一样

5.4.3 事务隔离级别

  1. 事务 未提交读
    • 可以读取到事务未提交时的数据
    • 相当于没有隔离
  2. 事务 提交读（事务过程中的数据不会被读到）
    • 事务修改过程中的数据不能读到，只能读到事务完成前后的数据。
    • 解决了，脏读
  3. 可重复读（不能修改其他事务已经读到的数据）
    • 读到的数据加锁（写锁，排他锁），不许其他事务修改。
    • 解决了，脏读、不可重复读
  4. 事务 串行化
    • 一个执行完，另一个再执行。
    • 解决了，脏读、不可重复读、幻读
